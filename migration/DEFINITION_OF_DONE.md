# Definition of Done — TraderX Migration

This checklist tracks completion of every constraint from `TARGET_ARCHITECTURE_CONSTRAINTS.md` across Process A (code migration) and Process B (infrastructure).

---

## Process A — Single-Tenant Migration & Service Extraction

### 1. Single-Tenant Isolation (TARGET_ARCHITECTURE_CONSTRAINTS.md Section 1)

- [x] App runs in single-tenant mode only
- [x] Tenant chosen at startup via `TENANT_ID` env var
- [x] Application fails fast if `TENANT_ID` is not set
- [x] Requests with mismatched `X-Tenant-ID` header rejected with 403
- [x] No shared in-memory state (`CURRENT_TENANT`, `_runtime_state`, `KNOWN_TENANTS` removed)
- [x] Tenant-specific business rules loaded per-tenant at startup, not from shared config dict
- [x] Frontend uses build-time `REACT_APP_TENANT_ID` — no runtime tenant switching
- [x] No `TenantContext`, `TenantProvider`, `fetchWithTenant`, or tenant selector in frontend code

### 2. Database-per-Tenant (TARGET_ARCHITECTURE_CONSTRAINTS.md Section 2)

- [x] Data isolation implemented (database per tenant)
- [x] `TENANT_ID` incorporated into default database connection string
- [x] `DATABASE_URL` env var overrides for production use
- [x] Two different `TENANT_ID` values produce two separate databases
- [x] `tenant_id` column no longer needed for row-level filtering
- [x] Tests still use in-memory or ephemeral database

### 3. Service Boundaries (TARGET_ARCHITECTURE_CONSTRAINTS.md Section 7)

- [x] Domain services extracted per service boundary table:
  - [x] Account Service (port 8001) — Account CRUD, account user management
  - [x] Trading Service (port 8002) — Trade submission, validation, state machine
  - [x] Position Service (port 8003) — Position tracking, queries, recalculation
  - [x] Reference Data Service (port 8004) — Stock ticker lookup, S&P 500 data
  - [x] People Service (port 8005) — Person directory, validation
- [x] Each service can be developed, deployed, and scaled independently
- [x] Each service owns its own data store (no shared database)

### 4. Cross-Service Communication (TARGET_ARCHITECTURE_CONSTRAINTS.md Section 8)

- [x] Cross-service communication uses HTTP APIs only (no shared DB access)
- [x] No direct imports between service packages
- [x] No circular dependencies between services
- [x] Inter-service URLs configured via environment variables
- [x] Dependency graph is a DAG (no cycles)

### 5. API Contracts

- [x] Each service has an OpenAPI spec (auto-generated by FastAPI at `/docs` or `/openapi.json`)
- [x] Each service has a `/health` endpoint returning `{"status": "UP", "service": "<name>", "tenant": "<TENANT_ID>"}`

### 6. Observability (TARGET_ARCHITECTURE_CONSTRAINTS.md Section 10)

- [x] Structured JSON logging with `tenant_id` and `service` fields
- [x] Correlation IDs included in log output

### 7. Graceful Shutdown

- [x] Each service handles `SIGTERM` gracefully (drain in-flight requests)

### 8. Frontend

- [x] Frontend API URLs point at extracted service ports (not the monolith)
- [x] Frontend has zero references to mutable tenant state
- [x] Frontend builds with `REACT_APP_TENANT_ID` and displays tenant correctly

### 9. Testing

- [x] Each service has its own test suite
- [x] All service test suites pass with `TENANT_ID` set
- [x] Integration test script exists and passes
- [x] Cross-service HTTP calls verified when all services running

### 10. Legacy Monolith

- [x] Original monolith directory preserved (not deleted)
- [x] Monolith README has deprecation notice pointing to `services/`
