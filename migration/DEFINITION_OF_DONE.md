# Definition of Done — TraderX Migration

This checklist tracks completion of every constraint from `TARGET_ARCHITECTURE_CONSTRAINTS.md` across Process A (code migration) and Process B (infrastructure).

---

## Process A — Single-Tenant Migration & Service Extraction

### 1. Single-Tenant Isolation (TARGET_ARCHITECTURE_CONSTRAINTS.md Section 1)

- [ ] App runs in single-tenant mode only
- [ ] Tenant chosen at startup via `TENANT_ID` env var
- [ ] Application fails fast if `TENANT_ID` is not set
- [ ] Requests with mismatched `X-Tenant-ID` header rejected with 403
- [ ] No shared in-memory state (`CURRENT_TENANT`, `_runtime_state`, `KNOWN_TENANTS` removed)
- [ ] Tenant-specific business rules loaded per-tenant at startup, not from shared config dict
- [ ] Frontend uses build-time `REACT_APP_TENANT_ID` — no runtime tenant switching
- [ ] No `TenantContext`, `TenantProvider`, `fetchWithTenant`, or tenant selector in frontend code

### 2. Database-per-Tenant (TARGET_ARCHITECTURE_CONSTRAINTS.md Section 2)

- [ ] Data isolation implemented (database per tenant)
- [ ] `TENANT_ID` incorporated into default database connection string
- [ ] `DATABASE_URL` env var overrides for production use
- [ ] Two different `TENANT_ID` values produce two separate databases
- [ ] `tenant_id` column no longer needed for row-level filtering
- [ ] Tests still use in-memory or ephemeral database

### 3. Service Boundaries (TARGET_ARCHITECTURE_CONSTRAINTS.md Section 7)

- [ ] Domain services extracted per service boundary table:
  - [ ] Account Service (port 8001) — Account CRUD, account user management
  - [ ] Trading Service (port 8002) — Trade submission, validation, state machine
  - [ ] Position Service (port 8003) — Position tracking, queries, recalculation
  - [ ] Reference Data Service (port 8004) — Stock ticker lookup, S&P 500 data
  - [ ] People Service (port 8005) — Person directory, validation
- [ ] Each service can be developed, deployed, and scaled independently
- [ ] Each service owns its own data store (no shared database)

### 4. Cross-Service Communication (TARGET_ARCHITECTURE_CONSTRAINTS.md Section 8)

- [ ] Cross-service communication uses HTTP APIs only (no shared DB access)
- [ ] No direct imports between service packages
- [ ] No circular dependencies between services
- [ ] Inter-service URLs configured via environment variables
- [ ] Dependency graph is a DAG (no cycles)

### 5. API Contracts

- [ ] Each service has an OpenAPI spec (auto-generated by FastAPI at `/docs` or `/openapi.json`)
- [ ] Each service has a `/health` endpoint returning `{"status": "UP", "service": "<name>", "tenant": "<TENANT_ID>"}`

### 6. Observability (TARGET_ARCHITECTURE_CONSTRAINTS.md Section 10)

- [ ] Structured JSON logging with `tenant_id` and `service` fields
- [ ] Correlation IDs included in log output

### 7. Graceful Shutdown

- [ ] Each service handles `SIGTERM` gracefully (drain in-flight requests)

### 8. Frontend

- [ ] Frontend API URLs point at extracted service ports (not the monolith)
- [ ] Frontend has zero references to mutable tenant state
- [ ] Frontend builds with `REACT_APP_TENANT_ID` and displays tenant correctly

### 9. Testing

- [ ] Each service has its own test suite
- [ ] All service test suites pass with `TENANT_ID` set
- [ ] Integration test script exists and passes
- [ ] Cross-service HTTP calls verified when all services running

### 10. Legacy Monolith

- [ ] Original monolith directory preserved (not deleted)
- [ ] Monolith README has deprecation notice pointing to `services/`

## Process B — Infrastructure (not in scope for Process A)

### Containerization (TARGET_ARCHITECTURE_CONSTRAINTS.md Section 3)

- [ ] Dockerfiles build successfully for all services
- [ ] Images tagged with semantic version and commit SHA
- [ ] Containers run as non-root users
- [ ] Health check endpoints defined in Docker `HEALTHCHECK`

### Kubernetes (TARGET_ARCHITECTURE_CONSTRAINTS.md Section 4)

- [ ] Kubernetes manifests or Helm charts valid
- [ ] Readiness and liveness probes configured
- [ ] Resource requests and limits defined
- [ ] Namespace isolation per environment

### CI/CD (TARGET_ARCHITECTURE_CONSTRAINTS.md Section 5)

- [ ] GitHub Actions CI pipeline functional
- [ ] Lint, type-check, unit tests, integration tests on PR
- [ ] Docker image build and push on merge
- [ ] Staging deployment automated
- [ ] Production deployment with manual approval gate

### Infrastructure as Code (TARGET_ARCHITECTURE_CONSTRAINTS.md Section 6)

- [ ] Terraform validates and applies successfully
- [ ] VPC, subnets, security groups defined
- [ ] Kubernetes cluster (EKS/AKS) provisioned
- [ ] Database instances per tenant provisioned
- [ ] Remote state backend with locking

### API Gateway (TARGET_ARCHITECTURE_CONSTRAINTS.md Section 9)

- [ ] API gateway routes requests to services by path prefix
- [ ] TLS termination configured
- [ ] Rate limiting per tenant

### Smoke Testing

- [ ] Smoke tests exist and pass against deployed environment
